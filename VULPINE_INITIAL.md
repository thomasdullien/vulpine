I would like to build a comprehensive vulnerability development agent called
"Vulpine".

The goal is to have set of agents and skills that I can run with different
backend models, both open and closed, and then evaluate their performance on
various vulnerability development tasks.

The different agents I would like to create are:

 1) Build preparation: An agent that takes a piece of software (usually a GitHub
   repository and optionally a commit hash for a specific version) and then
   prepares a working directory where this software can be
   built with ASAN, TSAN, and other sanitizer support, as well as function-level
   tracing support (use https://github.com/thomasdullien/cppfunctrace)

   Input: A link to a git repository with the source code of a project.
   Output: A directory with a dockerfile and a subdirectory with the source
   code. The dockerfile mounts the source code and the source code needs to 
   build cleanly with all sanitizers inside the container, and allow to be
   run easily. It needs to be trivial to run both sanitized and unsanitized
   versions and function-traced versions of the code.

 2) Code navigation preparation. The C/C++ codebase should be rebuilt with Bear
   (Build Ear) and then with Woboq (https://github.com/KDAB/codebrowser). 
   This provides human-readable browsable cross-referenced HTML output for the
   C/C++ codebase, but the data generated by Woboq can also be used for more
   effective C++ cross-reference navigation
   outside of pure sed and grep. There is a C++ command-line tool (here: ...)
   that is helpful in navigating large C/C++ codebases for agents.

   Input: The output of (1) - a directory with the dockerfile
   Output: An extra directory with the HTML files and indexing data from Woboq
   so that the code can be easily navigated.

 3) Attack surface modelling agent. This agent is supposed to read the project
   documentation, perform web searches to better understand the typical use case
   of the project and the deployment scenarios, and then examine the codebase
   to identify the set of features that should be reachable by the attacker in
   the codebase.

   Input: The source code directory and the command-line tool for code navigation
   that makes the cross-referenced code navigable.
   Output: An MD file called "ATTACK_SURFACE.md" which contains an enumerated
   list describing the important attack surfaces that are reachable by an
   attacker.

 4) Configuration agent. This agent is responsible for reading the project
   documentation and understanding how to configure the software in a typical
   setup for enterprise deployment, and to then generate a minimal testing
   container that resembles a real deployment enough to perform security
   analysis.

   Input: The source code directory, the command-line tool for code navigation,
   and an optional extra file called "CONFIGURATION.md" with special information
   about the desired configuration.
   Output: A bash script that takes the container built in (1) and creates all
   necessary data inside the container so that the software can be in a way
   that mimics a real-world deployment.

 5) Attack surface to code mapping agent: This agent takes each item of the
   attack surface described by agent (3) and generates simple deterministic
   fuzzers for the relevant feature. It then fuzzes these features while also
   generating coverage data for the relevant code. This coverage data is then
   used to identify areas of the code associated with specific functionalities,
   and the final output is a directory of files that map features identified
   in stage 3 to lists of functions uniquely associated to these features.

   Input: The source code directory, the command-line tool for code navigation,
   the command-line tool for processing function traces, and the dockerfile
   and container created by (1) and (4) and the ATTACH_SURFACE.md file.
   Output: A set of directories, one for each feature in the attack surface,
   each containing (A) a minimal deterministic fuzzer for that feature (B) 
   coverage data describing which functions were executed when that feature
   is exercised.

   Once this step is completed, this agent should launch many parallel (6) agents,
   one for each input file.

 6) Function audit log creator. A function audit log is a concise description
   of a C/C++ function or class method. It describes the following:
    - What was the programmers intent when implementing the function?
    - What is actually implemented?
    - Is there any discrepancy between the intent and the implementation? Are
      there any internal integer overflows, logic errors, omissions, or other
      footguns that lead to the function not always performing in the way that
      it was intended?
    - In which ways does this function modify global program state or the local
      class (in terms of free'ing memory or allocating memory or changing sizes)
      ? Is there any codepath that has lingering effects?
   The agent should populate a database of function audit logs so that for each
   function, class, or method we can easily look up what the code is supposed
   to do and what the quirks are.

   Input: The source code directory, the CLI tool for code navigation and
   for processing function traces, the dockerfile and container with the
   configured target, and the CLI tool to manage the function audit log data.
   Output: A populated database of function audit logs, so for any function
   in the codebase one can easily look up the summary of that function - and
   a list of functions that seem to have the most egregious misbehavior.

 7) Code auditor. Given the artifacts created by the other agents, begin
   carefully reading the code for security flaws. Loop through all features
   described by agent (3), use the attack surface to code mapping output (5)
   to identify the areas of the code associated with it, and begin studying
   it. Examine the function audit logs created in (6) carefully, and then see
   if there is any way that the program can be led into an "incoherent state",
   e.g. triggering a situation where the intent of the programmer and the
   actual actions diverge and we can violate programmer intent. These can
   lead to straight out-of-bounds memory access, but also to weird bugs like
   memory disclosure, confused deputy problems, shell escapes, race conditions,
   time-of-check-to-time-of-use issues, or other problems.
   When you have found such a situation, launch a subagent to attempt to build
   a minimal trigger for the condition.
   The output of this agent is a set of directories, each containing a report
   about the issue and a minimal reproducing trigger, and a gdb script that
   can be used to reliably verify that the trigger was hit.

   Input: The CLI tools for code navigation, function traces, function audit
   logs, the container with the target.
   Output: A subdirectory which containes one directory per confirmed issue,
   with a report of the underlying issue, the actual intended state that the
   program should've reached, what state was reached instead, and if this has
   any severe consequences (memory corruption, memory disclosure, other forms
   of confusion about the state). Along the report, we should have a minimal
   trigger program, and a GDB script that helps us demonstrate the trigger.

 8) Exploit developer. Given the outputs of all of the above, study the reports
   for the different bugs carefully. Think carefully about how these different
   bugs can be best combined to yield a reliable security exploit for the
   target.
   Build a hypothetic chain of events that describes each step along the way
   for a reliable exploit. Validate each step one-by-one. If any of the steps
   turns out to be impossible, try to see if the step can be replaced by a
   different step. With the output of (7), you will have a non-trivial number
   of different bugs, and they can be combined in creative ways.
   Keep iterating on this hypothetical chain, adjusting individual steps as
   you disprove them. If you come to the conclusion that the chain cannot work
   as described, document what you learnt in a concise MD file called
   "EXPLOIT_LEARNINGS.md", and then restart the exploitation process with a new
   chain. Keep updating EXPLOIT_LEARNINGS.md so that you can keep working even
   accross context limitations.
